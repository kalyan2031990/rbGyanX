1. Overview

rbGyanX is a radiobiology-guided clinical decision support framework designed for treatment plan evaluation in radiation oncology. It integrates:

  • Physical Dosimetry: Dose-volume histogram (DVH) processing and dose metric extraction
  • Radiobiological Modeling: TCP (Tumor Control Probability) and NTCP (Normal Tissue Complication Probability) models
  • Machine Learning: ANN and XGBoost models with SHAP explainability
  • Therapeutic Integration: UTCP, therapeutic window, and risk-benefit analysis
Note: rbGyanX is intended for research and academic use. It should be used as a decision support tool, not as a replacement for clinical judgment.

2. Workflow

The rbGyanX pipeline follows a standardized 6-step workflow:

Step 1 DVH Detection + Canonicalization
Preprocess and validate DVH files. Detects structure types (TARGET vs OAR) automatically.
Step 2 Physical Dose Metrics
Calculate physical dose metrics. OAR metrics (V5-V50, Dmax) and Target metrics (D95, D98, V95, V100, V107, HI, CI, GI).
Step 3 TCP and/or NTCP Modeling
Run traditional and/or ML models for TCP (targets) and/or NTCP (OARs) based on selected analysis mode.
Step 4 Clinical Factor Analysis
Analyze relationships between clinical factors and outcomes (requires clinical data).
Step 5 QA & Validation
Generate quality assurance reports (advisory only, non-blocking).
Step 6 TCP–NTCP Integration
Calculate therapeutic ratios (UTCP, P+, CFTC) when both TCP and NTCP are executed.

3. Input Requirements

3.1 DVH Files

rbGyanX accepts DVH files in multiple formats:

  • Format: CSV, TXT, or directory containing multiple DVH files
  • DVH Type: Cumulative or differential (auto-detected)
  • Required Columns: Dose[Gy] and Volume[cm³] or Volume[%]
  • Structure Names: Automatically detected (PTV/GTV/CTV = TARGET; others = OAR)
File Naming Convention: Files should be named with patient ID and structure name, e.g., Patient_001_PTV.csv or Patient_001_Parotid.csv

3.2 Clinical Data (Optional but Recommended)

Required for ML models and clinical factors analysis:

  • Format: Excel (.xlsx) or CSV (.csv)
  • Required Column: PatientID (must match DVH file naming)
  • Optional Columns: Age, Gender, TumorStage, Treatment, Outcome, etc.
  • Outcome Column: For supervised ML, include binary outcome (0/1 or No/Yes)

4. Analysis Modes

Select ONE analysis mode before starting:

Mode Description Requirements Outputs
TCP only Tumor Control Probability analysis Target structures (PTV/GTV/CTV) tcp_analysis/
NTCP only Normal Tissue Complication Probability analysis OAR structures (e.g., Parotid, SpinalCord) ntcp_analysis/
TCP + NTCP Both analyses with therapeutic integration Both target and OAR structures tcp_analysis/, ntcp_analysis/, integration/
Important: Mode selection determines which analysis branches run. Step 6 (integration) only activates in TCP + NTCP mode.

5. Step-by-Step Guide

5.1 Global Settings

  1. Select Analysis Mode (TCP only, NTCP only, or TCP + NTCP)
  2. Set Output Directory (all results will be saved here)
  3. (Optional) Browse for Clinical Data file (required for ML models)

5.2 Step 1: DVH Preprocessing

  1. Select input format (Single File or Directory)
  2. Browse and select your DVH input
  3. Select DVH type (Auto-detect recommended)
  4. Click "Run Step 1"
Step 1 automatically detects structure types (TARGET vs OAR) and validates DVH physics. Results are saved to processed_DVH/.

5.3 Step 2: Physical Dose Metrics

  1. Click "Run Step 2"
  2. Metrics are automatically calculated from Step 1 output

Outputs:

  • OAR metrics: dose_metrics/tables/NTCP_physical_metrics.xlsx
  • Target metrics: dose_metrics/tables/TCP_physical_metrics.xlsx
  • DVH plots: dose_metrics/plots/

5.4 Step 3: TCP/NTCP Analysis

Traditional Models Tab:

  1. Select which models to run (multiple selections allowed)
  2. Verify tumor site/diagnosis (auto-detected from DVH metadata)
  3. Select structures to analyze (populated after Step 1)

Edit Parameters Tab:

  1. Select a model from dropdown
  2. Adjust parameters if needed (defaults from literature)
  3. Click "Apply Changes" to save

ML Options:

  • Enable "ML Models" checkbox (requires clinical data)
  • Enable "SHAP Explainability" for feature importance analysis

Click "Run Step 3" to execute analysis.

5.5 Step 4: Clinical Factors Analysis

  1. Click "Run Step 4" (requires clinical data file)
  2. Analysis runs for both TCP and NTCP if both were executed
Clinical factors analysis identifies relationships between patient/clinical variables and outcomes.

5.6 Step 5: Quality Assurance

  1. Click "Run Step 5" (runs automatically after Step 4 if "Run All" was used)
  2. QA reports are generated (advisory only, non-blocking)
QA is advisory only. Warnings and flags do not indicate execution failure.

5.7 Step 6: TCP-NTCP Integration

  1. Only available when TCP + NTCP mode is selected
  2. Runs automatically after Step 5 completes (if "Run All" was used)
  3. Can be run manually by clicking "Run Step 6"

Outputs:

  • UTCP (Uncomplicated TCP) calculations
  • Therapeutic window analysis
  • Risk-benefit summary
  • Pareto frontier plots

8. Troubleshooting

Common Issues

Issue Solution
Step 1 fails with "No DVH files found" Check input path and file format. Ensure files contain Dose[Gy] and Volume columns.
Step 3 reports "No structures detected" Run Step 1 first. Verify structure names in DVH files match expected patterns (PTV/GTV/CTV for targets).
ML models not training Ensure clinical data file is provided and contains PatientID column matching DVH file names.
Step 6 integration not available Step 6 only runs in TCP + NTCP mode. Ensure both TCP and NTCP Step 3 completed successfully.
QA warnings appear QA is advisory only. Warnings do not indicate errors. Review QA reports for details.

Validation Errors

DVH validation in Step 1 may flag issues but does not block execution. Check:

  • Monotonicity: Dose should be ascending, volume should be descending (cumulative) or non-negative (differential)
  • Physical constraints: Doses and volumes should be positive and within expected ranges

9. Disclaimer and Usage Cautions

⚠️ Important Disclaimers

  • Research Tool Only: rbGyanX is intended for research and academic use. It should not be used as the sole basis for clinical decision-making.
  • Not a Medical Device: This software is not FDA-approved or certified as a medical device.
  • Clinical Validation Required: Model parameters and predictions should be validated against local clinical data before clinical use.
  • No Warranty: The software is provided "as is" without warranty of any kind.
  • User Responsibility: Users are responsible for ensuring appropriate use and interpretation of results.

Best Practices

  • Data Quality: Ensure DVH data is accurate and complete. Validate against treatment planning system.
  • Model Parameters: Use literature-based parameters as defaults. Custom parameters should be clinically validated.
  • ML Models: ML model performance depends on training data quality. Ensure adequate sample size and proper cross-validation.
  • Interpretation: Consider multiple models and clinical context when interpreting predictions.
  • Documentation: Document all analysis parameters and workflow steps for reproducibility.

Limitations

  • Models assume idealized dose distributions and may not account for all biological complexities
  • TCP/NTCP predictions are probabilistic and subject to uncertainty
  • ML models may exhibit overfitting with small datasets
  • Integration metrics (UTCP, P+) are simplified and should be interpreted with caution
For Support: Refer to the Help menu in rbGyanX or consult the documentation for detailed model descriptions and references.
